# Reverse proxy for microservices

This exercise moves from isolated containers into a more realistic distributed setup. The goal is to route traffic through a single entry point while multiple services run behind it. In practice, this is where Docker, Swarm, service placement, and a reverse proxy start to work together as an architecture instead of separate tools.

## What the stack includes

- A Swarm cluster with manager and worker nodes.
- A database service.
- Three application services.
- A reverse proxy service as the public entry point.
- An optional visualizer to inspect the cluster.

## Initialize the Swarm

Start on the manager node:

```bash
docker swarm init
```

Then join worker nodes with the token generated by the manager:

```bash
docker swarm join --token <WORKER_TOKEN> <MANAGER_IP>:2377
```

If you need to print the token again:

```bash
docker swarm join-token worker
```

## Label nodes by responsibility

Labels help you place services intentionally instead of leaving everything to a default scheduler decision.

```bash
docker node ls
docker node update --label-add db=true NODE_ID
docker node update --label-add worker=true NODE_ID
```

## Stack definition

The stack below shows a database, three services, a reverse proxy, and a visualizer. The important idea is not the exact image names, but how responsibilities are split and constrained.

```yaml
version: '3.8'

services:
  db:
    image: mongo:latest
    ports:
      - 27017:27017
    volumes:
      - db-data:/data/db
    deploy:
      placement:
        constraints:
          - node.labels.db == true
      resources:
        limits:
          cpus: '8.0'
          memory: 8G

  survey:
    image: rhonalchirinos/exec04-survey:latest
    ports:
      - 3000:3000
    command: sh -c "node /app/src/main.js"
    depends_on:
      - db
    deploy:
      replicas: 3
      resources:
        limits:
          cpus: '1.0'
          memory: 1G
      placement:
        constraints:
          - node.labels.worker == true

  auth:
    image: rhonalchirinos/exec04-auth:latest
    ports:
      - 3001:3001
    command: sh -c "node /app/src/main.js"
    depends_on:
      - db
    environment:
      - MONGO_DB=mongodb://db:27017/app
      - PORT=3001
    deploy:
      replicas: 3
      resources:
        limits:
          cpus: '1.0'
          memory: 1G
      placement:
        constraints:
          - node.labels.worker == true

  stats:
    image: rhonalchirinos/exec04-stats:latest
    ports:
      - 3002:3002
    depends_on:
      - db
    deploy:
      replicas: 2
      resources:
        limits:
          cpus: '1.0'
          memory: 1G
      placement:
        constraints:
          - node.labels.worker == true

  haproxy:
    image: rhonalchirinos/exec04-haproxy:latest
    ports:
      - "80:8080"
    depends_on:
      - stats
      - auth
      - survey
    deploy:
      placement:
        constraints:
          - node.role == manager

  visualizer:
    image: dockersamples/visualizer:latest
    ports:
      - "8080:8080"
    volumes:
      - "/var/run/docker.sock:/var/run/docker.sock:ro"
    deploy:
      placement:
        constraints:
          - node.role == manager
      replicas: 1

volumes:
  db-data:
```

## Deploy the stack

Once the images exist and the nodes are ready, deploy the stack from the manager.

```bash
docker stack deploy -c docker-compose.yml mystack
```

## Why the reverse proxy matters

The proxy becomes the public front door of the system. Instead of exposing every internal service directly, you centralize ingress and route requests according to hostnames, paths, or ports.

- The entry point stays controlled.
- Internal services can scale independently.
- You can move routing concerns into one place.
- The architecture becomes easier to evolve.

## What this exercise teaches

- Swarm can orchestrate multiple services with replica control.
- Node labels help separate infrastructure roles.
- A reverse proxy gives the system a stable external access point.
- Compose files for Swarm become a readable operational contract.

## A good next step

After the stack works, the next valuable improvement is to define explicit routing rules, health checks, and rolling updates. That is where the setup starts to look like a real production baseline.
